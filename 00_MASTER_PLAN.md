# Project VOX — Master Plan

A voxel world (Stone Age → end of Iron Age) populated by LLM-driven NPCs. Target: 10–30 rich agents on a single RTX A6000 (48 GB), Godot engine.

Companion docs: `QUICKSTART.md` (**how to run the game**), `01_TECH_TREE.md` (technology graph), `02_WORLD_ELEMENTS.md` (blocks/items/structures), `03_NPC_PERSONALITY.md` (agent psychology), `04_GAMEPLAY_ROADMAP.md` (**tech tree → playable systems plan, waves A–H**).

---

## 1. Architecture overview

```
┌────────────────────────┐        WebSocket (JSON events)        ┌──────────────────────────┐
│  GODOT 4.x (client+sim)│ ◄───────────────────────────────────► │  CORTEX (Python/FastAPI) │
│  - Voxel Tools terrain │   observations, actions, dialogue     │  - Agent loops (async)   │
│  - NPC bodies, pathing │                                       │  - Memory (SQLite+vector)│
│  - Behavior trees/GOAP │                                       │  - Prompt builder        │
│  - World/tech state    │                                       │  - Model router          │
└────────────────────────┘                                       └────────────┬─────────────┘
                                                                              │ OpenAI-compatible API
                                                              ┌───────────────┴───────────────┐
                                                              │  vLLM server(s) on A6000      │
                                                              │  main model + tiny model +    │
                                                              │  embedding model              │
                                                              │  (or any external API per NPC)│
                                                              └───────────────────────────────┘
```

Key principle: **the engine never talks to a model directly**. Cortex owns all cognition; Godot owns physics, rendering, and low-level behavior. Any LLM (local vLLM, llama.cpp, OpenAI/Anthropic API) can be **bound per NPC** via a config entry — this satisfies the "load or bind any LLM to each person" requirement.

### Per-NPC binding config
```yaml
npc: "kara_the_potter"
brain:
  provider: "vllm-local"          # vllm-local | llamacpp | anthropic | openai | ollama
  model: "qwen3-14b-awq"
  endpoint: "http://localhost:8000/v1"
  temperature: 0.8
  tier: "full"                    # full | dialogue-only | scripted
persona_file: "personas/kara.yaml"
memory_db: "memory/kara.sqlite"
```

## 2. Engine layer (Godot)

- **Godot 4.x + Voxel Tools (Zylann's `godot_voxel` module)** — C++ module, supports blocky (Minecraft-like) and smooth voxels, streaming, LOD. Use the blocky mesher with a 16³ chunk format.
- "Somewhat realistic" look: 0.5 m voxels for terrain, NPCs as voxel meshes (MagicaVoxel `.vox` imported via `godot-magicavoxel` importer), PBR-ish palette, ambient occlusion, day/night cycle.
- **World state is data-driven**: every block, item, station, and recipe is defined in JSON generated from `01_TECH_TREE.md` / `02_WORLD_ELEMENTS.md`. The tech tree is loaded as a DAG; a settlement's "known tech" set gates which recipes/structures NPCs can use.
- **Low-level NPC control stays in-engine**: locomotion, pathfinding (A* over voxel navgrid), and reactive behavior (flee fire, eat when starving) run as behavior trees / GOAP at 60 fps. The LLM issues *goals and utterances*, not keystrokes.

## 3. Cognition layer (Cortex service)

Generative-agents-style loop per NPC, but tiered so 10–30 agents stay affordable:

1. **Perceive** — Godot streams salient events ("saw wolf", "Toran gave me flint", time, needs).
2. **Memory stream** — every event stored with importance score + embedding (SQLite + FAISS/sqlite-vec). Retrieval = recency × importance × relevance.
3. **Reflect** — periodic LLM pass that compresses memories into beliefs ("Toran is generous").
4. **Plan** — daily plan + current goal, generated by LLM, decomposed into GOAP goals the engine can execute.
5. **Converse** — dialogue turns are full LLM calls with persona + relationship + retrieved memories in context.

**Tiering** (critical for scale): NPCs near the player or in a conversation run "full" (LLM plan + dialogue). Distant NPCs run "scripted" (behavior tree following their last daily plan). Promotion/demotion is automatic by distance and event salience. A tiny model handles background chatter and importance-scoring.

## 4. Model plan for the A6000 (48 GB)

> Verify latest releases at deployment time; this is the mid-2026-safe baseline.

| Role | Recommended | VRAM (quantized) | Notes |
|---|---|---|---|
| Main NPC brain (10–30 agents) | **Qwen3-30B-A3B (FP8/AWQ)** | ~18–30 GB | MoE, ~3B active params → high tokens/s, great at role-play + JSON tool calls |
| Alternative main | Qwen3-14B or Mistral-Small-24B (AWQ) | ~10–16 GB | Denser, slightly richer prose |
| Utility/background model | Qwen3-4B or Llama-3.2-3B | ~3 GB | Chatter, importance scoring, summarization |
| Embeddings (memory) | BGE-M3 or nomic-embed-text | ~1.5 GB | Runs alongside |

- Serve with **vLLM** (OpenAI-compatible, continuous batching — the thing that makes 30 concurrent NPC streams work). Remaining VRAM goes to KV cache; with a 14B-AWQ main model you get ~30 GB of KV cache, ample for 30 × 4k-token contexts.
- Godot rendering on a separate GPU or the iGPU if possible; otherwise cap vLLM `gpu_memory_utilization` at ~0.85.
- Structured output: use vLLM guided JSON for plans/decisions; free text only for dialogue.

## 5. Tech-tree integration rules

- The tech tree (see `01_TECH_TREE.md`) is a DAG of ~330 nodes across 8 eras. **Knowledge lives in NPCs, not in a global unlock bar**: a technology is "known" to a settlement when at least one living NPC has learned it (by teaching, observation, or discovery events).
- Every node lists the **world elements it requires and introduces**; `02_WORLD_ELEMENTS.md` is the reverse index. World-gen only spawns era-appropriate structures.
- Discovery = scripted opportunity events (lightning fire, copper nugget in hearth) + LLM-roleplayed "insight" when prerequisites and materials are present. This makes progression emergent but bounded.

## 6. Roadmap

| Phase | Scope | Exit criterion |
|---|---|---|
| P0 ✅ | Godot + Voxel Tools terrain, 1 walking voxel NPC | NPC pathfinds in a generated world |
| P1 ✅ | Cortex service + vLLM, 1 NPC full loop (perceive→plan→act→talk) | Chat with an NPC that remembers you |
| P2 ✅ | Needs/inventory/crafting engine, tech-tree JSON loader, era-1 content | NPC knaps a tool because it *decided* to |
| P3 ✅ | 10 NPCs, relationships, teaching/learning, tiering | Two NPCs exchange a technology |
| P4 ✅ | Survival core (fire, hunting, cooking, day/night, death), settlement dynamics, 30 NPCs | Village survives 10 in-game days unattended |
| P5 ✅ | Discovery engine, era tracking, E2 content wave, per-NPC model rebinding | Era transitions occur emergently |
| P6 ✅ | Generations (aging, births with trait inheritance, old-age death), seasons, focused discovery | Settlement advances eras across generational turnover |
| P7a ✅ | The living world: decoration layer, model trees, buildable huts (E2.20), wandering game, largest-component village placement | The village visibly grows and the world feels inhabited |
| P7b ✅ | World persistence (save/continue, per-map seeds), mood/emotion layer, E3-E4 wave (blades, sewing, bow) + fauna (hare/boar/fish) + wolf predators | The world endures restarts, feels, and bites back |
| P8 ✅ | Cortex flavors: vanilla vs EMERGENT (two 5-person villages, two families each, blank minds, zero narration — menu-selectable, hello-handshake roster switch, multi-fire world) | Two societies start from nothing, far apart |
| P8b+ | Gameplay waves A–H per `04_GAMEPLAY_ROADMAP.md`: storage → agriculture → pastoralism/crafts → metallurgy → trade → knowledge → (conflict?) | Iron Age reached emergently, with real fields, kilns and forges |
